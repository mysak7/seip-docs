Decision Framework for SEIP

SEIP Workload Profile

  • Event volume: low-to-medium (dev: few 100 events/min)
  • Burst pattern: bursty AI analysis → scale-to-zero needed
  • Data model: JSON events with TTL, claim-based processing
  • Teams: solo/small team — low operational overhead matters
  • Budget: dev environment — cost sensitivity high
  • Region: single region, no multi-AZ hard requirement for dev
  • Compliance: EU data residency preferred

What drives the alternatives analysis

  • Cosmos DB: serverless billing can spike unpredictably on dev loads
  • Event Hubs: minimum 1 TU cost even at zero events
  • AKS free tier: good for dev, but still needs node VMs
  • Azure OpenAI: no access to Claude, limited model choice
  • Managed Identity: Azure-only; breaks portability
  • Azure Monitor: powerful but expensive at higher ingestion

🗄️ Data Layer

Data Layer Alternatives

SEIP's data model is: JSON security events with TTL, a claim-based "lock" field for deep-mind workers, pattern storage, and cost records. The current stack uses DynamoDB (AWS) or Cosmos DB (Azure). Both charge per-request which can be unpredictable on dev workloads.

⚠️

Cosmos DB Serverless gotcha for SEIP: Each write costs ~1 RU/KB. With 200 events/min at 2KB each = ~400 RU/s. Serverless is billed at €0.282/million RU. Sounds cheap — but dev clusters with aggressive re-reads (KEDA polling, Deep Mind loops) can generate 50–100× more RUs than events. Always put a max RU cap on serverless or switch to the predictable alternatives below.

Alternatives to Cosmos DB / DynamoDB

Best Fit
🐘

PostgreSQL

RDS / Azure Database / Supabase / self-hosted

JSONB columns give full document storage with SQL queries. A single security_events table with data JSONB + status TEXT + created_at TIMESTAMPTZ replaces both DynamoDB tables. TTL via pg_cron or a scheduled DELETE.

Pros
  • Predictable cost — flat instance price
  • Rich SQL queries for seip-manager patterns
  • JSONB indexing (GIN) faster than GSIs
  • Self-hostable; zero lock-in
  • Supabase adds REST API for free
Cons
  • Schema migration needed from DynamoDB key model
  • Claim locking needs SELECT FOR UPDATE (not atomic like DynamoDB ConditionExpression)
  • Connection pooling (pgBouncer) needed at scale
  • Single-writer by default (no multi-region writes)
Good Fit
🍃

MongoDB Atlas

Serverless or M10 cluster / self-hosted

Document model is nearly identical to DynamoDB. Atlas Serverless has the same RU-like billing (Request Units) as Cosmos DB but with cleaner SDK. M10 dedicated at $57/mo gives flat cost and 10 GB storage — ideal for SEIP dev volume.

Pros
  • Zero schema migration — same JSON doc structure
  • Change Streams replace DynamoDB Streams
  • Atlas Search for event pattern queries
  • Multi-cloud (AWS + Azure) deployment
Cons
  • Serverless RU billing — same gotcha as Cosmos DB
  • M10 dedicated overkill for current scale
  • Third-party vendor (Atlas); not open-source
  • Claim atomicity: findOneAndUpdate is fine but less obvious
Good Fit
⚡

ScyllaDB

ScyllaDB Cloud / self-hosted on EC2

Cassandra-compatible, C++ rewrite. DynamoDB API compatibility layer available. Extremely low latency at high throughput — ideal if SEIP scales to millions of events/hour. For dev scale it is overkill, but makes sense if production targets 50K+ events/min.

Pros
  • DynamoDB API compatible — minimal code change
  • 10× lower cost vs DynamoDB at high throughput
  • Self-hostable with Terraform
  • Sub-millisecond p99 latency
Cons
  • Over-engineered for dev scale
  • Cluster ops complexity (multi-node recommended)
  • No serverless / scale-to-zero tier
  • DynamoDB compat layer not 100% feature-parity
Niche
🔴

Redis / Valkey

Elasticache / Azure Cache / Upstash / self-hosted

Not a primary store replacement, but Redis Streams can act as both the event queue AND a temporary event store (TTL natively). Upstash Redis is pay-per-command at $0.2/100K commands — cheaper than DynamoDB at SEIP's dev scale. Use alongside PostgreSQL, not as a standalone replacement.

Pros
  • Upstash: true zero-cost at low volume
  • Redis Streams = Kafka lite for SEIP
  • Native TTL on all keys
  • Atomic SETNX for claim locking
Cons
  • Memory-only — not a durable primary store
  • No complex queries without RediSearch
  • Data persistence needs AOF/RDB config
  • Single-purpose — needs a second DB for patterns
Niche
🦆

DuckDB + S3/Blob

Embedded analytics store

Not a transactional store — but excellent for seip-money cost analysis and seip-manager pattern queries that currently do DynamoDB scans. Embed DuckDB in the Python service, query Parquet files directly from S3/Blob. Replaces expensive full-table scans.

Pros
  • Zero infrastructure cost
  • SQL over Parquet on S3 — no ETL
  • Perfect for seip-money CUR/Cost reports
  • Runs in-process; no network calls
Cons
  • Read-only analytics — not for event ingestion
  • Not a DynamoDB replacement; complementary only
  • Concurrent writes not supported
AWS-Only
🔷

DynamoDB (keep on AWS)

Current AWS deployment

The existing seip-security-events and seip-patterns tables work well. PAY_PER_REQUEST with the claim-based lock is proven. If staying on AWS, no migration needed. The main risk is RCU/WCU spikes from KEDA polling the UnprocessedEvents count every 30s.

Pros
  • Zero migration cost
  • Atomic ConditionExpression for claim lock
  • DynamoDB Streams → Lambda for future fan-out
  • Single-digit ms latency at any scale
Cons
  • AWS vendor lock-in
  • KEDA polling causes steady RCU drain (~$2-5/day dev)
  • Scan operations expensive for analytics
  • No SQL; pattern queries need FilterExpression hacks

Migration Path: Cosmos DB → PostgreSQL

💡

PostgreSQL with JSONB is the strongest Cosmos DB replacement for SEIP. The claim-locking pattern, TTL, and JSON structure all map cleanly. Here's the target schema:

-- security_events table (replaces Cosmos DB security-events container)
CREATE TABLE security_events (
    id           TEXT PRIMARY KEY,                 -- partition key: eventId
    agent_id     TEXT NOT NULL,                    -- sort key: agentId
    data         JSONB NOT NULL,                   -- full event payload
    status       TEXT DEFAULT 'unprocessed',       -- claim field
    claimed_by   TEXT,                             -- deep-mind worker ID
    claimed_at   TIMESTAMPTZ,
    created_at   TIMESTAMPTZ DEFAULT now(),
    ttl_at       TIMESTAMPTZ                       -- pg_cron: DELETE WHERE ttl_at < now()
);

CREATE INDEX idx_events_status    ON security_events (status) WHERE status = 'unprocessed';
CREATE INDEX idx_events_agent     ON security_events (agent_id);
CREATE INDEX idx_events_data      ON security_events USING GIN (data);

-- Atomic claim lock (replaces DynamoDB ConditionExpression)
UPDATE security_events
   SET status = 'claimed', claimed_by = $1, claimed_at = now()
 WHERE id = $2
   AND status = 'unprocessed'           -- conditional, like ConditionExpression
RETURNING id;                           -- returns null if already claimed

-- seip-patterns table (replaces seip-patterns container)
CREATE TABLE patterns (
    id        TEXT PRIMARY KEY,
    name      TEXT NOT NULL,
    rule      JSONB NOT NULL,
    lua_hash  TEXT,
    updated_at TIMESTAMPTZ DEFAULT now()
);

⚡ Event Streaming Layer

Event Streaming Alternatives

Sysmon events from Windows agents flow via Fluent Bit to the event broker, then to seip-kafka-consumer. Currently: MSK/Confluent (AWS) or Azure Event Hubs. The consumer is standard confluent-kafka Python — the bootstrap server and credentials are the only config changes.

ℹ️

Key insight for SEIP: At dev scale (hundreds of events/min), the Kafka protocol overhead is the cost, not the data volume. All alternatives below speak native Kafka protocol — seip-kafka-consumer needs zero code changes, only env vars: KAFKA_BOOTSTRAP_SERVERS, KAFKA_SASL_USERNAME, KAFKA_SASL_PASSWORD.

Alternatives to Azure Event Hubs / MSK

Best Fit
🐼

Redpanda

Self-hosted on EC2/VM or Redpanda Cloud Serverless

Kafka-compatible broker written in C++ (no JVM). Single binary, runs on 1 vCPU / 512MB RAM. Deploy as a Docker container on the existing app VM alongside other SEIP services. Redpanda Cloud Serverless has a free tier (10 GB/month) and charges per GB ingested — at SEIP dev scale, likely $0/month.

Pros
  • Zero minimum cost self-hosted or Cloud free tier
  • Kafka-compatible: zero consumer code change
  • No JVM — runs in 200MB RAM on app VM
  • Redpanda Console UI built-in
  • 10× lower latency than Kafka
Cons
  • Self-hosted = you operate it
  • No SLA for free tier cloud version
  • Small ecosystem vs Apache Kafka
  • Single node = no replication (dev acceptable)
Good Fit
🚀

NATS JetStream

Self-hosted; nats.io open-source

Ultra-lightweight message broker with JetStream persistence layer. Not Kafka-compatible natively — seip-kafka-consumer needs a NATS Python client swap. But NATS server runs in 20MB RAM, has at-least-once delivery, consumer groups (Kafka-equivalent), and TTL on messages.

Pros
  • Lightest footprint — 20MB RAM server
  • Built-in KV store (replaces some Redis uses)
  • Consumer ack/nack = claim pattern natively
  • Zero external dependencies
Cons
  • Not Kafka-compatible — code change in consumer
  • Fluent Bit has no native NATS output plugin
  • Smaller community than Kafka
  • JetStream persistence has different semantics
Good Fit
☕

Apache Kafka (KRaft mode)

Self-hosted; no ZooKeeper since 3.3+

The reference implementation. KRaft mode (no ZooKeeper) since Kafka 3.3 makes self-hosting much simpler — single process. Still JVM-based (~500MB RAM). For SEIP, a single-node Kafka in KRaft mode gives production-compatible semantics at dev scale.

Pros
  • Gold standard — reference for all Kafka tooling
  • KRaft = no ZooKeeper, single binary now
  • Largest ecosystem; Kafka Streams, ksqlDB, Connect
  • Fluent Bit Kafka output is battle-tested
Cons
  • JVM: 500MB+ RAM overhead
  • Longer startup time (15–30s)
  • Config complexity vs Redpanda
  • Still needs disk for log storage
Niche
🌩️

Apache Pulsar

Self-hosted / StreamNative Cloud

Multi-tenant, geo-replicated streaming. Pulsar has a Kafka protocol adapter — seip-kafka-consumer works unchanged. Best use case: if SEIP needs to fan out events to multiple consumer groups independently (e.g., separate consumers per tenant Windows environment).

Pros
  • Kafka protocol adapter — zero consumer change
  • Tiered storage (S3/Blob) for cheap long retention
  • Multi-tenant isolation built-in
Cons
  • Heavy: needs BookKeeper + ZooKeeper
  • 3–5 services to operate; overkill for SEIP dev
  • Higher learning curve
No-Broker
🔗

Direct HTTP / gRPC

Fluent Bit HTTP output → FastAPI endpoint

Skip the broker entirely: Fluent Bit's http output plugin POSTs Sysmon events to a FastAPI endpoint in seip-kafka-consumer. The FastAPI endpoint validates and writes to the data store. Simplest possible architecture — appropriate for very low volume.

Pros
  • Zero broker infrastructure
  • Simplest debug path (curl to test)
  • No Kafka ops at all
Cons
  • No replay / retention — events lost if consumer down
  • No backpressure mechanism
  • Not scalable beyond single consumer
  • Loses Kafka topic fan-out capability

☸️ Kubernetes Layer

Kubernetes Platform Alternatives

EKS (AWS) and AKS (Azure) host the Orchestrator workloads with Karpenter/KEDA for autoscaling. For dev the control plane cost is the main concern: EKS costs $0.10/hr (~$73/mo) while AKS free tier is truly free but still needs worker node VMs.

Dev Winner
🪶

k3s

Rancher Labs — runs on a single VM

Lightweight certified Kubernetes distribution. Single 70MB binary, runs on the existing app VM (EC2 t4g.small / Azure D2as_v5). No external control plane cost. Full K8s API — Helm, KEDA, and Karpenter-lite via k3s node-driver all work. Perfect for SEIP dev.

Pros
  • Zero control plane cost
  • Runs alongside Docker services on app VM
  • Built-in Traefik ingress, CoreDNS, Helm
  • SQLite etcd (or external Postgres) built-in
  • Full kubectl / Helm compatible
Cons
  • No cloud-managed Karpenter — manual node scaling
  • Single node = no real HA for dev
  • Community support, not cloud SLA
  • Some CNI plugins need manual config
Cloud Managed
☁️

GKE Autopilot

Google Kubernetes Engine

GKE's managed Kubernetes with per-pod billing (not per-node). You pay only for the CPU/memory your Pods request — when scaled to zero, cost is zero. Strong Karpenter support. Good option if SEIP moves to GCP (Vertex AI is already used).

Pros
  • Pay-per-pod — true scale-to-zero cost
  • No node management at all
  • Collocates with Vertex AI (no cross-cloud LLM calls)
  • Workload Identity for GCP SA = IRSA equivalent
Cons
  • Third cloud provider — more accounts to manage
  • Data stays in GCP region (GDPR consideration)
  • Control plane cost: $0.10/hr (same as EKS)
  • Autopilot restricts some DaemonSet use cases
Current AWS
🟠

EKS + Karpenter

Current AWS deployment

Managed EKS control plane with Karpenter provisioning Spot EC2. The SEIP-specific NodePool with consolidationPolicy: WhenEmpty scales worker nodes to zero when no Pods run. EKS cost is $0.10/hr regardless of Pods; Karpenter nodes only bill when running.

Pros
  • Karpenter native — best autoscaler for AWS
  • Spot interruption handling built-in
  • IAM IRSA for KEDA permissions
  • CloudWatch metric trigger for KEDA
Cons
  • $73/mo control plane cost always-on
  • AWS lock-in for IAM + Karpenter
  • Two-step terraform apply (provider bootstrap)
Current Azure
🔵

AKS + Node Auto Provisioning

Current Azure deployment

AKS free tier means zero control plane cost. Node Auto Provisioning (NAP) uses the same Karpenter CRDs — NodePool and AKSNodeClass YAML is nearly identical to the AWS version. Best cloud-managed option for SEIP on Azure.

Pros
  • Free control plane tier
  • NAP = Karpenter CRDs, familiar YAML
  • Workload Identity for KEDA IRSA equivalent
  • Azure Monitor Container Insights built-in
Cons
  • NAP still preview in some regions
  • Spot VM interruption via Event Grid (more complex)
  • AKSNodeClass vs EC2NodeClass feature gap
Skip for SEIP
🔶

HashiCorp Nomad

Self-hosted task scheduler

Lighter alternative to Kubernetes. Native Docker support, built-in service mesh. No CRD complexity. However KEDA has no Nomad adapter — you would rebuild SEIP's autoscaling from scratch using Nomad's scaling stanzas and HCL policies.

Pros
  • Much simpler than K8s
  • Runs Docker containers natively
  • Single binary, low resource usage
Cons
  • No KEDA support — major rework needed
  • No Karpenter — cloud autoscaling manual
  • Smaller ecosystem; fewer Helm charts
  • HashiCorp BSL license change (2023)

📈 Autoscaling Layer

Autoscaling Alternatives

SEIP uses KEDA to scale the Orchestrator Pods 0→5 based on UnprocessedEvents in CloudWatch/Azure Monitor, and Karpenter/NAP to provision/deprovision EC2/VM nodes when Pods are scheduled. This is the most nuanced layer — the trigger source can be swapped independently of the scaling mechanism.

💡

Separate the concerns: Pod autoscaling (KEDA, HPA, Knative) and Node autoscaling (Karpenter, Cluster Autoscaler, NAP) are independent. KEDA is the right answer for pod scaling in SEIP regardless of cloud. The node scaler and trigger source are what changes per environment.

Pod Autoscaler Options

Best Fit
🎯

KEDA (keep it)

Kubernetes Event-Driven Autoscaler v2.16

KEDA is purpose-built for SEIP's pattern: scale from zero to N based on an external queue depth. The trigger type changes per environment but the operator and ScaledObject YAML stays the same. Swap cloudwatch → azure-monitor → postgresql → prometheus by changing one field.

Pros
  • Scale-to-zero natively — HPA can't do this
  • 40+ trigger sources out of the box
  • PostgreSQL trigger: query COUNT(*) WHERE status='unprocessed'
  • Prometheus trigger: read from Grafana stack
  • Works on any K8s: k3s, EKS, AKS, GKE
Cons
  • Polling interval adds latency to scale-up
  • IRSA/Workload Identity needed per trigger source
  • TriggerAuthentication setup per environment
Built-in K8s
📊

HPA + VPA

Horizontal + Vertical Pod Autoscaler

Standard Kubernetes autoscalers. HPA scales on CPU/memory or custom metrics (via metrics-server + Prometheus adapter). VPA right-sizes resource requests. Limitation for SEIP: HPA cannot scale to zero — minimum replica count is 1, which means a running Deep Mind pod even when idle.

Pros
  • Zero extra installation — built into K8s
  • VPA auto-tunes resource requests
  • No external metric source needed (CPU-based)
Cons
  • Cannot scale to zero — always ≥1 pod
  • CPU-based scaling doesn't match SEIP's queue depth model
  • Custom metrics need Prometheus Adapter setup
Serverless K8s
🌊

Knative Serving + Eventing

CNCF serverless layer for Kubernetes

Knative scales HTTP-based services from 0→N based on concurrent requests. Knative Eventing can subscribe to Kafka/NATS topics and trigger scale-up. For SEIP's Deep Mind pattern (pull-based, not HTTP), Knative Eventing is the right subproject — but adds significant complexity.

Pros
  • True serverless scale-to-zero
  • Knative Eventing: CloudEvents standard
  • Blue-green traffic splitting built-in
Cons
  • Complex: Istio or Kourier required
  • Pull-based workers need Eventing (not Serving)
  • SEIP Deep Mind is not HTTP-triggered
  • Heavy: adds 5–10 system pods

Node Autoscaler Options

AWS
⚙️

Karpenter v1

AWS EC2-native node provisioner

Provisions exact EC2 instance types for pending Pods within seconds. WhenEmpty consolidation removes idle nodes. SEIP's NodePool targets t3.micro/t3.small Spot in private subnets, tagged for isolation from the Deep Mind ASG.

Pros
  • Fastest provisioning (~60s)
  • Spot interruption handling via SQS
  • EC2NodeClass maps to SEIP's existing VPC
Cons
  • AWS-only
  • IAM roles needed (IRSA)
  • OIDC endpoint bootstrap dependency
Azure
🔵

AKS Node Auto Provisioning

Karpenter for Azure (NAP)

Same Karpenter CRDs as AWS (NodePool, NodeClaim) but with AKSNodeClass instead of EC2NodeClass. Spot VM interruption via Azure Event Grid → Service Bus. Consolidation policy identical.

Pros
  • Same YAML as AWS Karpenter
  • No separate install (AKS built-in)
  • Workload Identity for auth
Cons
  • NAP still preview in some regions
  • AKSNodeClass has fewer options than EC2NodeClass
Classic
🔧

Cluster Autoscaler

Cloud-agnostic; works on EKS, AKS, GKE

The original Kubernetes cluster autoscaler. Scales node groups (ASG/VMSS) up/down based on pending Pods and under-utilization. Slower than Karpenter (2–5 min scale-up vs 60s), and cannot scale to zero (ASG minimum=1). Use only if Karpenter is not available.

Pros
  • Works on all clouds
  • Stable, mature, well-documented
  • No extra CRDs
Cons
  • Minimum 1 node always running
  • 2–5 min scale-up latency
  • No WhenEmpty consolidation
🤖 LLM / AI Layer

LLM & AI Alternatives

seip-models (port 8184) routes LLM calls. Currently: AWS Bedrock (Claude on AWS), Azure OpenAI (GPT-4o on Azure), plus Vertex AI/Gemini which stay unchanged across both clouds. The model router pattern means swapping providers is a config change, not a code change.

ℹ️

SEIP's LLM workload: Security event triage — classify Sysmon events as benign/suspicious/critical. Needs strong reasoning for false-positive reduction. Token volume is low (few KB per event) but quality matters. Latency tolerance: 5–30 seconds per event analysis is acceptable.

LLM Provider Alternatives

Best Fit
🧠

Anthropic API Direct

Claude 3.5 Sonnet / Claude 3 Haiku

Call Anthropic's API directly — no AWS or Azure dependency. Claude 3.5 Sonnet is the best model for security event reasoning today. Claude 3 Haiku for fast triage, Claude 3.5 Sonnet for deep analysis. seip-models already has an Anthropic client; this removes the Bedrock intermediary.

Pros
  • Best reasoning for security triage
  • No cloud infrastructure dependency
  • Haiku: $0.25/M input tokens — cheapest quality option
  • Prompt caching: 90% token savings on repeated system prompts
  • Works from any cloud or local dev
Cons
  • Not EU-hosted — data leaves region (GDPR consideration)
  • Third-party API key management needed
  • No VPC-private endpoint (unlike Bedrock)
  • Rate limits on free/low-tier accounts
Privacy-First
🏠

Ollama (local / in-cluster)

LLaMA 3.3, Qwen2.5, Mistral — on-prem inference

Run open-weight models on the app VM or a GPU node. Ollama serves an OpenAI-compatible API — seip-models can call it with just a base URL change. For SEIP's security use case, models like llama3.3:70b or qwen2.5:32b on a g-series VM give adequate quality for Sysmon triage. No data leaves the VPC.

Pros
  • Data never leaves VPC — ideal for sensitive events
  • Zero per-token cost after hardware
  • OpenAI-compatible API — minimal code change
  • Runs on ARM64 (t4g.medium or larger)
Cons
  • Quality gap vs Claude/GPT-4o on complex events
  • GPU VM cost ($0.50–2/hr) can exceed API cost
  • Model download (10–40GB) on first run
  • Latency: 10–60s for 70B models on CPU
AWS Option
🟠

AWS Bedrock (Claude)

Current AWS deployment

Bedrock gives VPC-private access to Claude via a VPC endpoint. IAM-auth (IRSA) — no API key needed. Claude 3.5 Sonnet available. The main limitation: not available on Azure, so seip-models needs conditional routing on cloud.

Pros
  • VPC-private endpoint — no internet egress
  • IAM auth via IRSA — no key rotation
  • Claude 3.5 Sonnet + Haiku available
  • AWS Cost Explorer integration
Cons
  • AWS-only — breaks Azure portability
  • Bedrock per-token pricing ~same as direct API
  • Latency slightly higher vs direct API
  • Model availability varies by region
Already Used
🌐

Vertex AI / Gemini

Google Cloud — already in seip-models

Gemini 2.0 Flash and Pro are already in seip-models and work identically on AWS and Azure. Gemini 2.0 Flash is extremely fast and cheap ($0.075/M input tokens). For Sysmon event triage, Flash is often sufficient — keep Sonnet for complex multi-event correlation.

Pros
  • Already deployed in seip-models
  • Cheapest capable model for high-volume triage
  • Long context window (1M tokens)
  • Cloud-agnostic (direct GCP API, no VPC needed)
Cons
  • Third GCP account to manage
  • Flash: quality gap on complex security reasoning
  • Pro: cost comparable to Claude Sonnet
Azure Option
🔵

Azure OpenAI

GPT-4o / GPT-4o-mini — current Azure deployment

Azure-hosted OpenAI models via Managed Identity (no API key). GPT-4o is strong for security analysis but lacks the specialized reasoning that Claude offers for complex event chains. GPT-4o-mini is the cost-saving fallback, comparable to Gemini Flash in quality.

Pros
  • Managed Identity auth — no key management
  • EU data residency (germanywestcentral)
  • Private endpoint available
  • GPT-4o strong for structured output (JSON events)
Cons
  • No Claude on Azure — weaker reasoning
  • Quota approval needed for GPT-4o Turbo
  • Azure-only
Open-Source
🤗

vLLM + HuggingFace Models

Self-hosted on GPU node in EKS/AKS

🔑 Identity Layer

Identity & Secrets Alternatives

SEIP uses IAM IRSA (AWS) or Managed Identity + Workload Identity (Azure) for credential-free pod authentication. Secrets (API keys, Lua signing key) live in SSM Parameter Store (AWS) or Key Vault (Azure). The main pain: this is cloud-specific and breaks portability.

Cloud-Agnostic
🏛️

HashiCorp Vault

Self-hosted / HCP Vault (managed)

Central secrets manager that runs anywhere. Vault's Kubernetes Auth method lets pods get short-lived tokens via the K8s service account token — identical pattern to IRSA/Workload Identity, but works on k3s, EKS, AKS, and locally. HCP Vault Dedicated ($0.03/hr) for managed option.

Pros
  • Works on all clouds + local dev
  • Dynamic secrets: Vault generates DB passwords per pod
  • Audit log for every secret access
  • Vault Agent Injector for sidecar injection
  • Transit Engine replaces KMS for Lua signing
Cons
  • Operational overhead — must manage Vault cluster
  • HA Vault needs Raft backend (3 nodes)
  • HCP Vault adds $22+/mo cost
  • Initial setup complexity (policies, auth methods)
K8s Native
🔐

External Secrets Operator

Syncs cloud secrets into K8s Secrets

ESO pulls secrets from any provider (AWS SSM, Azure Key Vault, GCP Secret Manager, Vault) and creates native Kubernetes Secrets. Pods just mount standard K8s Secrets — no SDK change. The secret backend is abstracted. Best option when keeping cloud-native backends but wanting a unified K8s API.

Pros
  • Unified K8s Secret API regardless of backend
  • Works with SSM, Key Vault, Vault simultaneously
  • Secret rotation: ESO re-syncs on TTL
  • No SDK changes in SEIP containers
Cons
  • Secrets land in etcd (unless ETCD encryption enabled)
  • Still needs cloud identity (IRSA/Workload Identity) for the ESO pod
  • Extra Helm chart to manage
Zero-Trust
🆔

SPIFFE / SPIRE

CNCF workload identity framework

SPIFFE assigns cryptographic X.509 SVIDs to workloads by their identity (not their network location). SPIRE is the reference implementation. Each SEIP microservice gets a unique identity like spiffe://seip/deep-mind. mTLS between services is automatic. Replaces both Managed Identity and API keys for inter-service auth.

Pros
  • True zero-trust between microservices
  • Works on VMs + Kubernetes + containers
  • Auto-rotates certificates (short-lived SVIDs)
  • No cloud dependency
Cons
  • High complexity — mTLS in all seip-* services
  • Requires SPIRE server + agents deployed
  • Overkill for single-region dev environment
Simple
🔒

Sealed Secrets (Bitnami)

Encrypted K8s secrets in Git

Sealed Secrets lets you commit encrypted secrets to Git safely. The SealedSecret controller decrypts them using a cluster-private key at deploy time. Good for small teams — secrets live in the same repo as YAML manifests. No external secret store needed.

Pros
  • Secrets-as-code in Git
  • Zero external dependency
  • Simple: one controller, kubeseal CLI
  • Works on k3s with no cloud identity
Cons
  • No dynamic secrets or rotation
  • Cluster key loss = all secrets lost
  • No audit log
  • Manual re-seal when rotating secrets
📊 Observability Layer

Observability Alternatives

Currently: CloudWatch Logs + Metrics (AWS) or Azure Monitor + Log Analytics (Azure). Both are powerful but expensive at scale and cloud-specific. KEDA reads from these metrics sources — switching observability also means updating the KEDA trigger source.

Best Fit
📈

Grafana + Prometheus + Loki

Open-source observability stack (LGTM)

The canonical open-source observability stack. Deploy all three in-cluster on k3s or as sidecars. Prometheus scrapes SEIP Python metrics (Flask's /metrics endpoint). Loki collects Docker/Pod logs. Grafana visualizes everything. KEDA prometheus trigger reads directly from Prometheus — no cloud needed.

Pros
  • Zero cost — runs in-cluster
  • KEDA Prometheus trigger: cloud-agnostic
  • Custom dashboards for SEIP-specific metrics
  • Works identically on AWS, Azure, k3s, local
  • Grafana Alerting replaces CloudWatch Alarms
Cons
  • Storage: Prometheus TSDB on disk — needs PV
  • Long retention needs remote_write (Thanos/Mimir)
  • 3 extra Helm releases to manage
  • Loki label cardinality limits
Simple SaaS
🔭

Grafana Cloud Free Tier

Managed Prometheus + Loki + Tempo

Grafana Cloud's free tier: 10K active series (Prometheus), 50GB logs/month (Loki), 50GB traces/month (Tempo). SEIP's dev scale fits easily within the free tier. No infrastructure to run — push from in-cluster agents. KEDA needs a remote-read endpoint or a local Prometheus scraping into Cloud.

Pros
  • Free tier covers SEIP dev
  • No storage management
  • Tempo tracing out-of-the-box
  • Cloud-agnostic: push from anywhere
Cons
  • Data leaves VPC (GDPR consideration)
  • KEDA still needs local Prometheus or cloud metric
  • Free tier limits (10K series can fill up)
Current Azure
🔵

Azure Monitor + Log Analytics

Current Azure deployment

Azure Monitor receives custom metrics from SEIP via the Azure Monitor Ingestion API. Log Analytics Workspace for container logs (Container Insights). KEDA azure-monitor trigger reads the UnprocessedEvents custom metric directly. Tight integration with AKS diagnostic settings.

Pros
  • Deep AKS integration
  • Container Insights auto-installs with AKS
  • KQL queries for complex log analysis
  • Managed Identity auth — no keys
Cons
  • Cost: $2.30/GB ingested — expensive at scale
  • Azure-only; breaks portability
  • Custom metric ingestion needs SDK + MI
All-in-One
🐶

Datadog

Commercial observability platform

Full-stack observability SaaS. Datadog Agent runs as a DaemonSet on AKS/EKS nodes. Excellent Kubernetes and container insights. KEDA has a Datadog trigger. Cost scales linearly with hosts — SEIP's scale (1–3 nodes) fits the free trial easily but production cost is ~$23/host/month.

Pros
  • Best UI for complex investigation
  • KEDA Datadog trigger available
  • APM + logs + metrics in one product
  • Excellent Sysmon log parsing rules available
Cons
  • Expensive: $23+/host/month
  • Data leaves VPC (no on-prem option at this tier)
  • Vendor lock-in
Log-Focused
🔍

OpenSearch (self-hosted)

Open-source Elasticsearch fork

OpenSearch + OpenSearch Dashboards replaces the ELK stack (now fully open-source Apache 2.0). Index Sysmon events directly for ad-hoc threat hunting. Complements Prometheus — metrics stay in Prometheus, security events go to OpenSearch for analyst dashboards.

Pros
  • Full-text search on security events
  • Threat hunting: regex across all Sysmon fields
  • Apache 2.0 — no BSL/AGPL restrictions
Cons
  • Heavy: 2GB+ RAM for single node
  • Not for metrics (use Prometheus instead)
  • Adds another persistent volume requirement

Recommended Stack for SEIP (All Environments)

💡

Run Prometheus + Grafana + Loki in-cluster via kube-prometheus-stack Helm chart. This single chart installs everything. KEDA uses the prometheus trigger pointing to the cluster-local Prometheus. No cloud account needed for observability. Expose Grafana via port-forward or Ingress for the dashboard.

# Add to seip-infrastructure/k8s/observability/

helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --set prometheus.prometheusSpec.retention=7d \
  --set grafana.adminPassword=changeme

# KEDA ScaledObject with Prometheus trigger (no cloud needed)
apiVersion: keda.sh/v1alpha1
kind: ScaledObject

  name: orchestrator-scaledobject

  scaleTargetRef:
    name: seip-orchestrator

📦 Container Registry Layer

Container Registry Alternatives

SEIP builds 12 Docker images via GitHub Actions and pushes to ECR (AWS) or ACR (Azure). Currently the registry pull is authenticated via IAM IRSA / Managed Identity — no image pull secrets in pods.

Best Fit
🐙

GitHub Container Registry (GHCR)

ghcr.io — free for public + private repos

GHCR is already where GitHub Actions runs. Push during CI/CD with the GITHUB_TOKEN — no extra credentials. Private images are free for org repos. Works on any Kubernetes cluster via an imagePullSecret or by making images public. Eliminates the ECR/ACR dependency entirely for dev.

Pros
  • Free for private repos on GitHub
  • Zero setup — GITHUB_TOKEN works in Actions
  • Cloud-agnostic: pull from k3s, EKS, AKS equally
  • Linked to GitHub repo for audit trail
  • Packages UI shows image layers and digests
Cons
  • imagePullSecret needed in K8s (unlike IRSA/MI)
  • No scan-on-push (need separate Trivy step)
  • No lifecycle policies (images never auto-delete)
  • Bandwidth: 1GB/month free; then $0.08/GB
Self-Hosted
⚓

Harbor

CNCF project — self-hosted registry

Enterprise-grade self-hosted registry with vulnerability scanning (Trivy built-in), RBAC, and policy enforcement. Deploy on the app VM or as a K8s deployment. Stores images in S3/Blob backend. Replication: push to GHCR and replicate to Harbor, or vice versa.

Pros
  • Images stay in VPC
  • Vulnerability scanning built-in (Trivy)
  • Content trust + image signing
  • S3/Blob backend = cheap storage
Cons
  • Ops overhead — Harbor is a complex service
  • PostgreSQL + Redis + core services needed
  • Overkill for solo/small team
  • HA setup requires 5+ pods
Current AWS
🟠

AWS ECR

Current AWS deployment

Private registry in the same region as EKS. IAM pull via IRSA — no imagePullSecret in pods. Scan on push via Inspector. Lifecycle policies auto-clean old images. Cost: $0.10/GB-month storage + standard data transfer.

Pros
  • IRSA auth — no imagePullSecret
  • Lifecycle rules for auto-cleanup
  • Cross-account replication
  • VPC endpoint for private pulls
Cons
  • AWS-only
  • Region-scoped; cross-region pull costs
  • Token refresh every 12h needed for long-running creds
Current Azure
🔵

Azure Container Registry (ACR)

Current Azure deployment

Standard SKU ACR with Managed Identity pull. AKS attaches the ACR via --attach-acr — no imagePullSecret needed. Defender for Containers for vulnerability scanning. Geo-replication available (Premium SKU).

Pros
  • Managed Identity pull — no imagePullSecret
  • Defender scan on push
  • Private endpoint support
  • Retention policies via lifecycle rules
Cons
  • Azure-only
  • Standard SKU: $0.167/day + $0.003/GB storage
  • Geo-replication only on Premium ($0.667/day/replica)
Public Images
🐳

Docker Hub

Registry for base images only

Not recommended for SEIP application images — Docker Hub's free tier has rate limits (100 pulls/6h unauthenticated). However, SEIP's base images (python:3.12-slim, node:20-alpine) come from Docker Hub. Use a pull-through cache in ECR/ACR to avoid rate limits in CI.

Pros
  • Universal — all base images are here
  • Free for public images
Cons
  • Rate limits hit CI pipelines
  • No private images on free tier
  • Use ECR/ACR pull-through cache instead

Recommended GitHub Actions Push Pattern (GHCR)

# .github/workflows/build.yml
- name: Log in to GHCR
  uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}  # No extra secrets needed

- name: Build and push seip-orchestrator
  uses: docker/build-push-action@v5
  with:
    context: ./seip-orchestrator
    push: true
    tags: ghcr.io/${{ github.repository_owner }}/seip-orchestrator:${{ github.sha }}
    cache-from: type=gha
    cache-to: type=gha,mode=max

# K8s imagePullSecret (one-time setup for private GHCR images)
kubectl create secret docker-registry ghcr-secret \
  --docker-server=ghcr.io \
  --docker-username=mysak7 \
  --docker-password=$(gh auth token) \
  --namespace seip
