Azure System Overview

SEIP on Azure uses Azure Event Hubs (Kafka-compatible) for event ingestion, Cosmos DB for NoSQL storage, AKS with Node Auto Provisioning (Karpenter for Azure) for Kubernetes workloads, KEDA with Azure Monitor triggers for event-driven scaling, and Azure Managed Identity everywhere — replacing IAM roles and instance profiles.

ℹ️

Target region: germanywestcentral (Frankfurt) — Azure's closest equivalent to AWS eu-central-1. All services have data residency in Germany and comply with EU data sovereignty requirements.

Core Azure Services

🌐

Azure Virtual Network

10.0.0.0/16 VNet with Bastion subnet, public subnet, and two private subnets across two AZs. NAT Gateway for outbound, NSGs for traffic control, Private Endpoints for all PaaS.

☸️

AKS + NAP + KEDA

AKS cluster with Node Auto Provisioning (Karpenter for Azure) provisioning Spot VMs on demand. KEDA watches Azure Monitor custom metric UnprocessedEvents and scales pods 0→5.

⚡

Azure Event Hubs

Kafka-compatible event streaming. Windows Fluent Bit agents connect with zero code change — only the bootstrap server and SASL credentials change. Standard tier, 1 Throughput Unit.

🗄️

Cosmos DB for NoSQL

Serverless tier replaces DynamoDB PAY_PER_REQUEST. Two containers: security-events and seip-patterns. Built-in TTL, composite indexing replaces GSIs.

🖥️

App VM + VMSS

Standard_D2as_v5 (ARM64) app VM hosts all Docker services. Standard_B2s VMSS (0–5) for Deep Mind workers, autoscaled via Azure Monitor custom metrics.

📦

Azure Container Registry

Standard-tier ACR replaces ECR. Managed Identity pull — no credentials in containers. Scan-on-push via Microsoft Defender for Containers. Retention policies via lifecycle rules.

🔑

Managed Identity + Key Vault

User Assigned Managed Identity for app VM. AKS Workload Identity (OIDC) replaces IRSA for KEDA and app pods. Key Vault stores secrets and the Lua signing key.

🤖

Azure OpenAI + Multi-LLM

Azure OpenAI (GPT-4o, GPT-4o-mini) added as a "bridge" provider in seip-models. Vertex AI and Gemini direct API remain unchanged. Bedrock dependency eliminated.

Key Differences from AWS

What changes

  • fck-nat → NAT Gateway + Azure Bastion: managed outbound NAT, browser-based SSH
  • DynamoDB → Cosmos DB: NoSQL API, partition key model is compatible
  • ECR → ACR: Managed Identity pull, no image pull secrets
  • IAM Roles/IRSA → Managed Identity + Workload Identity
  • CloudWatch → Azure Monitor + Log Analytics
  • EventBridge + SQS → Event Grid + Service Bus (for Karpenter NAP interruptions)
  • Bedrock → Azure OpenAI (added as bridge provider)
  • SSM Run Command → Azure Run Command
  • S3 → Blob Storage (Lua filters, public container)
  • CUR → Cost Management API (seip-money integration)

What stays the same

  • All 12 Docker containers: unchanged images, same ports
  • Kafka consumer code: only bootstrap server changes
  • Karpenter NodePool/NodeClass YAML: same CRDs on AKS NAP
  • KEDA operator + ScaledObject pattern: trigger type changes
  • Terraform structure: modules pattern preserved
  • Fluent Bit on Windows agents: output plugin config only
  • Vertex AI / Gemini: no change in seip-models
  • Lua filter logic: identical Lua scripts
  • DynamoDB claim mechanism: replicated in Cosmos DB
  • GitHub Actions OIDC: federated identity, different audience

⚠️

The app VM needs a Standard public IP for port 8765 (seip-mcp). In AWS this was handled by fck-nat DNAT. On Azure, assign the public IP directly to the app VM's NIC and allow 8765 in the NSG. The NAT Gateway handles all other outbound traffic.

AKS, Node Auto Provisioning & KEDA

AKS with Node Auto Provisioning (NAP) is the Azure equivalent of EKS + Karpenter. NAP uses the same Karpenter CRDs — AKSNodeClass and NodePool — so the Kubernetes manifests are nearly identical. KEDA watches Azure Monitor instead of CloudWatch.

AKS Cluster Design

System Node Pool

  • VM SKU: Standard_D2s_v5 (2 vCPU, 8 GiB)
  • Count: min=1, max=3, desired=2
  • Zone: zones 1 + 2
  • Hosts: CoreDNS, kube-proxy, Karpenter/NAP controller, KEDA
  • Label: node-role: system
  • No spot: system pool must be regular priority

Workload Nodes (NAP-provisioned)

  • VM SKUs: Standard_D2s_v5, Standard_D4s_v5 (flexible)
  • Priority: Spot (evictable)
  • Count: 0 to max 10 (scales from zero)
  • Hosts: seip-orchestrator pods
  • Label: karpenter.sh/nodepool: workload
  • Interruption: Event Grid → Service Bus → NAP controller

AKSNodeClass (replaces EC2NodeClass)

apiVersion: karpenter.azure.com/v1alpha2
kind: AKSNodeClass
metadata:
  name: default
spec:
  imageFamily: AzureLinux        # Microsoft's minimal container OS
  osDiskSizeGB: 32
  # NAP discovers subnets tagged with karpenter.sh/discovery=seip-dev-aks
  # (same tag convention as Karpenter on EKS — apply via Terraform aws_ec2_tag equivalent)

NodePool (identical schema to EKS Karpenter)

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: workload
spec:
  template:
    metadata:
      labels:
        node-role: workload
    spec:
      nodeClassRef:
        apiVersion: karpenter.azure.com/v1alpha2
        kind: AKSNodeClass
        name: default
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot"]          # Azure Spot VMs — same cost benefit as EC2 Spot
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["Standard_D2s_v5", "Standard_D4s_v5", "Standard_D2as_v5"]
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30s
  limits:
    cpu: "40"
    memory: "80Gi"

KEDA — Azure Monitor Trigger

The cloudwatch trigger from the AWS version becomes azure-monitor. The Deep Mind VMSS publishes UnprocessedEvents as a custom Azure Monitor metric using the azure-monitor-ingestion SDK. KEDA reads that metric and scales the Orchestrator Deployment 0→5.

apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: keda-azure-monitor-auth
  namespace: seip
spec:
  podIdentity:
    provider: azure-workload   # Uses AKS Workload Identity (replaces IRSA)
---
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: seip-orchestrator-scaler
  namespace: seip
spec:
  scaleTargetRef:
    name: seip-orchestrator
  minReplicaCount: 0
  maxReplicaCount: 5
  pollingInterval: 30
  cooldownPeriod: 300
  triggers:
    - type: azure-monitor
      metadata:
        resourceURI: subscriptions/<SUB_ID>/resourceGroups/seip-dev-rg/providers/Microsoft.Compute/virtualMachineScaleSets/seip-dev-deep-mind-vmss
        tenantId: <TENANT_ID>
        subscriptionId: <SUB_ID>
        resourceGroupName: seip-dev-rg
        metricNamespace: seip/deepmind       # Custom metric namespace
        metricName: UnprocessedEvents
        aggregationType: Average
        targetValue: "30"                    # Same target as AWS: 30 events/replica
        metricAggregationInterval: "0:1:0"  # 1-minute window
      authenticationRef:
        name: keda-azure-monitor-auth

Publishing Custom Metrics from Deep Mind

# In seip-deep-mind: replace boto3 CloudWatch call with Azure Monitor Ingestion
# requirements: azure-monitor-ingestion azure-identity

from azure.monitor.ingestion import LogsIngestionClient
from azure.identity import DefaultAzureCredential
import datetime

def publish_unprocessed_count(count: int):
    # Azure Monitor custom metrics via Metrics Advisor endpoint (REST)
    # Alternative: use azure-mgmt-monitor CustomMetricsClient
    import requests, json
    from azure.identity import DefaultAzureCredential

    credential = DefaultAzureCredential()
    token = credential.get_token("https://monitoring.azure.com/.default").token

    url = (
        "https://germanywestcentral.monitoring.azure.com"
        "/subscriptions/<SUB_ID>"
        "/resourceGroups/seip-dev-rg"
        "/providers/Microsoft.Compute/virtualMachineScaleSets/seip-dev-deep-mind-vmss"
        "/metrics"
    )
    body = {
        "time": datetime.datetime.utcnow().isoformat() + "Z",
        "data": {
            "baseData": {
                "metric": "UnprocessedEvents",
                "namespace": "seip/deepmind",
                "dimNames": [],
                "series": [{"dimValues": [], "sum": count, "count": 1}]
            }
        }
    }
    requests.post(url, json=body, headers={"Authorization": f"Bearer {token}"})

EKS Cleanup Script → AKS Cleanup Script

#!/usr/bin/env bash
# cleanup_aks.sh — drain AKS workload nodes before terraform destroy
set -euo pipefail
CLUSTER="seip-dev-aks"
RG="seip-dev-rg"
SUB="<SUBSCRIPTION_ID>"

az aks get-credentials --resource-group "$RG" --name "$CLUSTER" --overwrite-existing

echo "=== Scaling KEDA ScaledObjects to 0 ==="
kubectl delete scaledobject --all -n seip 2>/dev/null || true
kubectl scale deployment seip-orchestrator --replicas=0 -n seip 2>/dev/null || true

echo "=== Deleting NAP NodePools ==="
kubectl delete nodepool --all 2>/dev/null || true

echo "=== Waiting for Karpenter/NAP nodes to terminate ==="
for i in $(seq 1 30); do
  COUNT=$(kubectl get nodes -l karpenter.sh/nodepool --no-headers 2>/dev/null | wc -l || echo 0)
  [ "$COUNT" -eq 0 ] && break
  echo "  $COUNT NAP nodes remaining — waiting 10s ($i/30)"
  sleep 10
done

echo "=== Destroying AKS cluster only ==="
terraform destroy -target=module.aks_cluster \
  -var-file=environments/dev-aks/terraform.tfvars \
  -auto-approve

echo "Done. VNet, Cosmos DB, Blob Storage, and VMSS are untouched."

Identity & Security

Azure replaces IAM roles + instance profiles with Managed Identities. IRSA becomes AKS Workload Identity. SSM Parameter Store becomes Key Vault. The result is a credential-free environment where no access keys exist in any container or VM.

Managed Identities Overview

🖥️

App VM Identity

User Assigned MI: seip-dev-app-mi. Granted: ACR Pull, Cosmos DB Data Contributor, Blob Storage Contributor, Key Vault Secrets User, Event Hubs Data Receiver.

⚙️

VMSS (Deep Mind) Identity

User Assigned MI: seip-dev-worker-mi. Granted: ACR Pull, Cosmos DB Data Contributor, Azure Monitor Metrics Publisher. Same as app VM minus GUI and management permissions.

☸️

AKS Control Plane MI

System Assigned MI: manages VNet integration, load balancer, and NAP subnet discovery. Needs Network Contributor on VNet subnets.

🔧

AKS Kubelet MI

User Assigned MI: seip-dev-kubelet-mi. Granted: ACR Pull on seipdevacr. All node pulls use this identity — no imagePullSecrets needed in Pod specs.

Azure Monitor Alerts (replaces CloudWatch Alarms)

Cosmos DB High RU Alert

Fires when TotalRequestUnits exceeds 3000/min sustained for 5 minutes. Replaces the DynamoDB ConsumedReadCapacityUnits alarm.

UnprocessedEvents High Alert

Fires when SEIP/DeepMind/UnprocessedEvents exceeds 500 for 10 consecutive minutes. Indicates a stuck or overloaded Deep Mind worker pool.

VMSS Scale Events

Activity Log alert on VMSS scale-out/scale-in events. Replaces CloudWatch scaling activity notification.

AKS Node Pool Events

seip-money Integration with Azure Cost Management

The seip-money service currently reads CUR reports from S3. On Azure, it queries the Azure Cost Management API directly — no export file needed. The REST API returns cost breakdowns by service, resource group, and tag.

# seip-money: replace S3 CUR reader with Azure Cost Management API call
# Uses DefaultAzureCredential (Managed Identity on app VM)

from azure.identity import DefaultAzureCredential
from azure.mgmt.costmanagement import CostManagementClient
import datetime

def get_monthly_costs(subscription_id: str) -> dict:
    credential = DefaultAzureCredential()
    client = CostManagementClient(credential)
    scope = f"/subscriptions/{subscription_id}"

    result = client.query.usage(scope, {
        "type": "ActualCost",
        "timeframe": "MonthToDate",
        "dataset": {
            "granularity": "Daily",
            "aggregation": {"totalCost": {"name": "Cost", "function": "Sum"}},
            "grouping": [{"type": "Dimension", "name": "ServiceName"}]
        }
    })
    return result.as_dict()

Migration Plan

Seven phases, runnable in parallel where dependencies allow. AWS and Azure run simultaneously during the cutover phase — no big-bang migration. Estimated elapsed time for a single engineer: 3–4 weeks.

Phased Roadmap

Phase 1 — Foundation Week 1

Create Resource Group, VNet with all subnets, NAT Gateway, Azure Bastion, User Assigned Managed Identities, Key Vault, GitHub OIDC federated credential. No application change. Terraform state: environments/dev/.

Phase 2 — Data Services Week 1–2

Deploy Cosmos DB (serverless), Azure Event Hubs (Kafka), Blob Storage (Lua + Cost exports), ACR. Run DynamoDB → Cosmos DB data migration script. Validate seip-patterns and security-events containers.

Phase 3 — App VM Week 2

Provision Standard_D2as_v5 VM with User Assigned MI. Update cloud-init script to pull from ACR (via MI, no credentials). Point containers at Cosmos DB and Event Hubs endpoints. Validate seip-mcp on port 8765 with public IP.

Phase 4 — Deep Mind VMSS Week 2

Deploy VMSS with autoscale profile. Wire seip-deep-mind to publish UnprocessedEvents to Azure Monitor. Validate scale-out and scale-in at threshold=30. Confirm Cosmos DB claim mechanism works.

Phase 5 — AKS + NAP + KEDA Week 3

Deploy AKS cluster with system node pool. Install KEDA via Helm with Workload Identity annotation. Apply AKSNodeClass and NodePool manifests. Apply ScaledObject with azure-monitor trigger. Validate scale-from-zero.

Phase 6 — Code Changes Week 3

Add Azure OpenAI as bridge provider in seip-models. Update seip-money to use Cost Management API. Update Fluent Bit config on Windows agents (Event Hubs endpoint). Update GitHub Actions to use Azure federated login.

Phase 7 — Cutover & Decommission Week 4

Update DNS for seip-mcp to Azure IP. Run parallel validation (both AWS + Azure for 48h). Switch Fluent Bit on all agents to Event Hubs. Run terraform destroy on AWS after 48h clean run. Tag Azure resources with cost center.

Risks & Mitigations

HIGH

DynamoDB → Cosmos DB Data Migration

security-events table may have millions of items. Full scan + write risks hitting Cosmos DB serverless burst limits and takes time. The analyzing_by claim mechanism must be replicated exactly.

Mitigation

Use AWS Data Migration Service + custom Python script. Migrate in chunks with TTL items filtered. Test claim mechanism on Cosmos DB before cutover. Keep DynamoDB as read replica during parallel run.

HIGH

No Claude on Azure OpenAI

seip-deep-mind and seip-manager use Claude via Bedrock for LLM analysis. Azure OpenAI only offers GPT-4o. Prompt engineering and output format expectations may differ, affecting classification accuracy.

Mitigation

Keep Vertex AI (Gemini) as the primary model — it's already supported and has no AWS dependency. Azure OpenAI is additive. Bedrock calls can remain cross-cloud via VNet egress if needed during transition.

MED

Event Hubs Kafka Compatibility Edge Cases

Event Hubs is ~95% Kafka compatible. Some Confluent-specific APIs (AdminClient.create_topics, consumer group management via Admin API) are not supported.

Mitigation

Review consumer_aws.py for any AdminClient usage. Topics are created via Terraform (Event Hub resources) rather than Kafka AdminClient — this aligns with Event Hubs' approach.

MED

AKS NAP (Karpenter) Maturity Gap

AKS Node Auto Provisioning is GA since 2024 but less battle-tested than EKS Karpenter. Some advanced Karpenter features (budget disruption, PodDisruptionBudgets interaction) may behave differently.

Mitigation

Start with WhenEmpty consolidation (conservative). Test scale-from-zero manually before going live. Monitor NAP controller logs in Log Analytics for provisioning errors.

LOW

Azure Monitor Custom Metrics Publishing

Replacing boto3 CloudWatch.put_metric_data with the Azure Monitor REST API requires adding azure-identity to seip-deep-mind requirements.txt and writing the custom metrics publisher.

Mitigation

Minimal code change — publish_unprocessed_count() is self-contained. DefaultAzureCredential picks up the VMSS Managed Identity automatically. Test locally with az login before deploying.

Verdict

When Azure wins for SEIP

  • No EKS control plane cost — AKS is free (saves ~$72/mo)
  • Managed Identity — cleaner than IAM instance profiles + SSM
  • Azure Bastion — no bastion EC2 instance to maintain
  • NAT Gateway — fully managed vs fck-nat EC2 instance
  • Event Hubs — first-class Kafka without MSK overhead
  • Organisation already on Azure — unified billing, AD integration

When AWS wins for SEIP

  • Claude via Bedrock — Azure OpenAI lacks Claude, best model for SEIP analysis
  • DynamoDB maturity — more predictable than Cosmos DB serverless burst
  • Karpenter maturity — EKS Karpenter more battle-tested than AKS NAP
  • Existing Terraform — already written and working
  • MSK/Confluent — full Kafka API compatibility vs Event Hubs 95%
💡

Recommendation: Migrate only if there is an organizational reason (Azure EA, existing AD, compliance requirement). The Claude/Bedrock dependency is the biggest practical blocker — seip-deep-mind's AI quality depends on it. A hybrid approach (Azure infra + cross-cloud Bedrock via NAT Gateway egress) is viable during a transition period.
